# ExecPlanについて
複雑な機能や大規模なリファクタリングに取り組む際は、必ず [plans.md](./plans.md) を使用してください。
「ExecPlan」という言葉を使ったときは、[plans.md](./plans.md) を参照し、以下を実行してください：
1. 全体像を理解する
2. 進捗状況を確認する
3. 作業後に [plans.md](./plans.md) を更新する
4. 発見事項と決定事項を記録する

[plans.md](./plans.md) はあなたの長期記憶であり、プロジェクトの羅針盤です。

## 原則
- 文書は日本語で書いてください。
- テスト駆動開発（TDD）を徹底する
- any, unknown 型を使用しない
- as ではなく satisfies を使用する
- 型定義はできるだけ詳細に行う
- 行動終了時に以下の3点を必ず確認する
  1. `npm run tsc`
  2. `npm run lint`
  3. `npm run test`
  - すべて成功することを確認し、問題があれば対応について検討し、修正を繰り返す
- Next.js + TypeScript + Vitest で実装する
- Next.jsはappディレクトリ構成を採用する
- TDDかつSDDで開発する
- テストファイルと実装ファイルは同じディレクトリに置く
- テストには極力モックが入らないようにする
- 変数名は自己説明的な名前とし、コメントが不要なようにする
- 関数は単一責任の原則に従う
- 作業が完了したらドキュメントを更新してください。
- 確認が完了したら、変更をコミットしてください。

## App Router の生成規約
- page/layout は Server Component を既定とし、"use client" はインタラクションを担う葉コンポーネントのみに限定する
- データアクセスは app/lib/data/ 配下の DAL から行い、UIへ渡すのは DTO のみとする
- ミューテーションは app/lib/actions/ 配下の Server Action を経由する
- 共有ロジックは Custom Hook へ寄せ、Container コンポーネントでデータ取得をしない
- if 文と早期 return を避け、分岐は式（例: 三項演算子）で表現する

## 実装手順
### テスト作成
t-wada氏によるTDD手順に従う
1. .agent/spec/を参照する
2. 実装前に実装予定のコンポーネントの作成ディレクトリと同じ場所で並ぶように `実装予定のコンポーネント.spec.ts` を作成する
3. `実装予定のコンポーネント.spec.ts` にコメントアウトで仕様を書き出す。（テストリストの作成）
4. `実装予定のコンポーネント.spec.ts` でテストコードを書く
5. 期待する結果は実装がないため失敗することを確認する

### 実装
1. 実装を行う
2. ci.shを実行し、型チェック・lint・テスト・ビルドの確認を行う
3. すべて成功するまで2を繰り返す
4. [.plans.md](./plans.md)に思考履歴を書き残す
5. 1〜4を繰り返す
